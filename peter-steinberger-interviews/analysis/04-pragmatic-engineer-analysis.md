> 采访者：Gergely Orosz (The Pragmatic Engineer)
> 采访时间：2026年1月（约1.5小时深度对话）
> 视频链接：https://www.youtube.com/watch?v=TQPYG87E6Nw

这是 Peter Steinberger 迄今为止最深入、最全面的一次采访，涵盖了他从 PSPDF Kit 到 Clawdbot 的完整职业历程，以及他对 AI 辅助开发的系统性思考。

## 一、Peter 的创业历程：从奥地利农村到硅谷

### 早年经历与编程启蒙

Peter 来自奥地利农村（rural Austria），14岁时接触到第一台电脑，从此迷上编程。他的第一个"项目"是给一个从学校偷来的 DOS 游戏写软盘拷贝保护程序，然后拿去卖。这种"先做出来再说"的黑客精神贯穿了他整个职业生涯。

**关键时间节点 [00:02:01 - 00:05:07]**：
- 14岁开始编程
- 大学期间需要自己赚学费（来自贫困家庭，从未见过父亲）
- 第一份正式工作被要求学习 Microsoft MFC（"我还在做噩梦"），但他偷偷用 .NET 2.0 重写
- 看到 iPhone 后"摸了一分钟就立刻买了一台"

### 第一个 iOS App 的诞生 [00:05:29 - 00:08:52]

Peter 的第一个 iOS app 是一个约会应用客户端。起因令人意外：

> [00:05:32] "我在地铁上用一个约会应用，写了很长一条消息，按发送时刚好进隧道，JavaScript 禁用了发送按钮，弹出错误消息，但那时候没有复制粘贴，没有截图，也不能滚动了。我的消息就这么没了，我气炸了。回家就下载了 Xcode。"

他用正则表达式解析 HTML（"完全不应该这么做"），使用 iPhone OS 3 beta、Core Data beta、以及一个破解版的 GCC 编译器来支持 blocks 语法。第一个月赚了 $10,000。

**值得关注的洞察**：
- 愤怒可以是创业的强大动力
- 不需要完美的技术栈，能运行就行
- 先做出来，用户会告诉你需要什么

### PSPDF Kit 的诞生 [00:09:35 - 00:12:50]

Peter 被介绍给一个需要修复"杂志阅读器"的项目，打开代码发现：

> [00:10:14] "这是我见过的最烂的代码。字面意思是一个几千行的 Objective-C 文件，还用窗口当标签页用。我都不知道这能跑起来。"

他花了两个月重写，朋友问他要代码，他就把 PDF 部分抽出来卖。用 WordPress 模板改了改放在 GitHub Pages 上，用 Dropbox 链接发送源代码压缩包。第一周卖了三份，每份 $200。

**更重要的是**：收到 10 封邮件抱怨功能不够。这是典型的"nerd sniped"（被技术问题吸引）：

> [00:12:43] "他们说没有文本选择功能。我想，能有多难？三个月后... 真的很难。"

**关于 PDF 格式的复杂性 [00:13:48 - 00:15:13]**：

有个客户发来一个 50,000 页的加拿大法律文本，每页超过 100 个链接，总计超过 50 万个链接。Peter 原本的数据模型完全崩溃，因为他的假设偏差了 1000 倍。花了两个月重新设计内部结构，同时保持 API 兼容。

**商业智慧 [00:17:55 - 00:19:12]**：

> [00:17:16] "如果你建的东西每个开发者都想自己做，那就很难卖。卖开发者工具本来就难。但如果是'天哪我不想做这个'加上'天哪这太难了'，那就是好位置。"

他的营销策略：只关心开发者，不做冷邮件，全靠口碑。如果能说服公司里的开发者，他们会帮你做内部推销。

### 职业倦怠与三年休息期 [00:28:24 - 00:32:09]

公司成长到 70 人时，Peter 卖掉了股份（现在公司接近 200 人）。

**倦怠的真正原因 [00:28:56 - 00:29:49]**：

> [00:28:56] "倦怠不一定来自工作太多。更多是当你工作在一件不再相信的事情上，或者有太多冲突。我们管理团队经常吵架，那时候我错误地认为公司应该更民主化地管理。"

休息期间的状态：
- 有几个月完全没开电脑
- 经历了"我该做什么"的存在危机
- "你不应该这么早退休，或者有这么好的退出，不用再工作。这搞乱了我的心理。"

## 二、重返技术：AI 工具的"圣杯时刻"

### 错过早期 AI 工具反而成为优势 [00:32:35 - 00:34:51]

2025 年 4 月，Peter 结束三年休息，想用 Web 技术重建一个推特分析项目。他完全不懂 React：

> [00:33:59] "什么是 prop？就是那种程度。这是很多开发者的陷阱——你在一种技术上越精通，跳到其他地方就越痛苦。"

他直接从 Claude Code 开始，跳过了所有早期 AI 工具：

> [00:34:54] "我要感谢那三年没碰电脑，因为在那段时间，你们这些人试过 AI 然后学会了它是垃圾。"

### 第一次"Holy Shit"时刻 [00:35:54 - 00:37:05]

他把一个 1.3MB 的 Markdown 文件（整个 GitHub 仓库转换的）拖进 Google AI Studio，输入"write me a spec"，得到 400 行规范。然后把规范拖进 Claude Code，说"build"，然后不断按"continue"...

> [00:36:46] "它告诉我'100% production ready'，我启动它，崩溃了。"

但关键是他加了一个浏览器 MCP，让它自己调试，几个小时后他有了一个能用的推特登录页面：

> [00:37:03] "那一刻，我看到了潜力。我理解了这是往哪个方向走的。"

### AI 编程的"老虎机效应" [00:38:40 - 00:39:44]

> [00:38:40] "这和去赌场一样的经济学原理。就是我的小老虎机。你按下去，叮叮叮叮叮，然后——不行。你输入 prompt，它要么做出垃圾，要么做出让你惊叹的东西。就是这种..."

Gergely 问：你是一个非常有经验的开发者，不容易被惊艳吧？

## 三、核心原则："闭环"（Closing the Loop）

### 为什么 AI 擅长写代码 [00:58:31 - 01:00:12]

这是整个采访中最重要的概念之一：

> [00:58:03] "你总是让它写测试，并确保它运行测试。"
> 
> [00:58:31] "这就是为什么当前的模型这么擅长写代码，但在创意写作上只是一般般——因为没有简单的方式验证写作是否好，但代码可以编译、可以 lint、可以执行、可以验证输出。"

**实际操作示例 [00:58:06 - 00:58:58]**：

他在调试一个 Mac 应用无法找到远程网关的问题。Mac 应用调试很烦——要构建、启动、手动测试。他告诉 agent：

> "你要建一个专门用于调试的 CLI，调用所有相同的代码路径，然后你自己迭代修复。"

Agent 跑了一个小时，回来说："这里有个竞态条件，这里有个配置错误，blah blah blah"。他不需要看代码。因为测试通过了。

### 闭环让你成为更好的架构师 [01:00:50 - 01:01:10]

> [01:00:50] "使用 agentic coding 实际上让你成为更好的程序员，因为你必须更努力地思考架构，使其更容易验证。因为验证是让事情做好的方法。"

**具体做法**：

1. 即使是网站，他也设计成可以通过 CLI 运行核心逻辑
2. 浏览器循环太慢，要有快速执行的反馈循环
3. 设计功能时就要想"怎么闭环？模型要能验证自己的工作"

### 端到端测试示例 [00:59:37 - 01:00:23]

> [00:59:37] "我告诉 Codex：设计端到端测试，启动 Docker 容器，安装整个系统，启动循环，用我这个文件里的 API key，然后让模型读取图片、创建图片、再看图片里有什么。不仅测试工具调用，还测试它能不能工作。"

Agent 花了很长时间，但测试了所有 API key（Anthropic、OpenAI、GLM 等），修复了所有工具调用和顺序问题。

## 四、Agentic Engineering vs Vibe Coding

### 术语澄清 [00:58:32 - 00:59:00]

> [00:58:32] "我不喜欢 vibe coding 这个词。vibe coding 现在几乎是个贬义词了。我叫它 agentic engineering（代理工程），带个星号。Vibe coding 从凌晨 3 点开始。"

**区别在于**：
- Vibe coding：随便试试，凌晨 3 点做的事情会后悔
- Agentic engineering：有系统地与 agent 协作，有明确的验证循环

### 与模型对话而非命令 [00:59:59 - 01:01:27]

> [01:00:00] "你不是告诉它做什么。你和模型对话。'我们看看这个，有什么选项？你考虑过这个功能吗？'"

关键技巧：
- 每个 session，模型都从对你的产品零了解开始
- 给它一些指引，让它探索不同方向
- 不需要 plan mode——对话到你说"build"才会开始构建
- 触发词：用"let's discuss"或"give me options"，它就不会开始构建

### 感受模型的"摩擦力" [00:40:13 - 00:41:21]

> [00:40:21] "有人说'你写代码时能感受到摩擦力，这就是你做出好架构的方式。'我 prompting 时感受到同样的摩擦力——我看代码飞过，看它花多长时间，看 agent 是否 push back，看它创建的东西是乱七八糟还是有道理。"

他已经能预测 prompt 需要多长时间。如果花的时间比预期长很多，就知道哪里出问题了。

## 五、Claude Code vs Codex (GPT 5.2) 对比

### Codex 的优势 [00:42:29 - 00:43:45]

> [00:42:43] "我不知道为什么这些人还在用 Claude Code... OpenAI 做的东西非常非常好。基本上我输入的每个 prompt 都能得到我想要的结果，这是疯狂的。"

**具体对比 [00:43:00 - 00:43:45]**：

| 特性 | Claude Code | Codex (GPT 5.2) |
|-----|-------------|-----------------|
| 文件阅读 | 读 3 个文件就开始写代码 | 静默读文件 10 分钟 |
| 速度 | 更快但需要多次迭代 | 慢 10 倍但几乎总是一次成功 |
| 需要的引导 | 需要推动它读更多代码 | 自动深入理解代码库 |
| 并行使用 | 因为快所以适合单终端 | 适合 5-10 个 agent 并行 |

### 如何有效使用 Claude Code 的技巧

> [00:43:17] "如果你很习惯 Claude Code 的工作方式，你需要忘掉很多... 那些你用来从 Claude Code 获得好输出的蠢办法。"

关于 Ralf Wigum one-liner（一种流行的提示技巧）：

> [00:48:53] "这又是人们用来绕过 Opus 模型限制的蠢概念。用 Codex 你根本不需要这个。"

## 六、代码评审已死？PRs = Prompt Requests

### 对 Pull Requests 的新看法 [01:10:30 - 01:11:35]

> [01:10:32] "我现在把 pull requests 更多看作 prompt requests。有人开 PR，我不逐行看代码，我说谢谢，思考这个功能，然后和我的 agent 一起从 PR 开始重新设计。"

**他的实际流程**：

1. 收到 PR，感谢贡献者
2. 让模型 review，提出问题
3. 重新设计功能，按自己的视角
4. Agent 很少直接复用代码，更多是获得功能目标的理解
5. 基本上每个 PR 都重写

> [01:11:12] "很多人 vibe code，整体 PR 代码质量下降很多。构建成功的功能仍然需要对整体设计的深刻理解。"

### 他现在想要什么 [01:14:04 - 01:14:42]

> [01:14:06] "我实际上更想要 prompts 而不是代码。我让人们把 prompts 加到 PR 里，我读 prompts 比读代码更多。"

为什么？

> [01:14:13] "这给我更高的信号——你是怎么得到这个方案的？你实际问了什么？涉及多少引导？这比实际输出告诉我更多关于输出的信息。"

### 对小 fix 的态度

> [01:14:47] "有人发只有几个修复的 PR。我告诉他们别这样。我 review 那个需要 10 倍的时间，不如我直接在 Codex 里输入'fix'然后等几分钟。"

## 八、公司如何适应 AI

### 30% 的人就够了 [01:23:55 - 01:24:38]

> [01:23:55] "我可以轻松用 30% 的人运营一个公司。找这样的人可能很难，但你想要真正资深的工程师，真正理解他们在构建什么，同时也舒服于委派任务，知道哪些部分重要需要专注，哪些部分可以 vibe。"

### 公司需要"重构"——不只是代码

> [01:24:12] "公司很难有效采用 AI。这也需要完全重新定义公司的运作方式。"

**Google 的例子**：
- Google 说你要么是工程师要么是经理
- 如果你想定义 UI 的样子，那个角色不存在

> [01:25:15] "非常可怕，因为经济上这会导致灾难。很多人会很难在新世界找到位置。"

### 为 Agent 优化代码库

> [01:26:15] "我设计代码库不是为了对我方便，而是为了对 agent 方便。不总是我喜欢的方式，但是我知道对这些模型最有效、摩擦最小的方式。因为我只想更快，最终是它们要处理代码，不是我。"

## 九、Clawdbot：让技术"消失"

### 超个人化 AI 助手的愿景 [01:28:03 - 01:30:50]

从 2025 年 4 月开始，Peter 就想要：

> [01:28:15] "一个真正深度理解我的助手。不是那种发早安邮件'这是你的三个任务'的。而是比如我见朋友，回家时它 ping 我'那个会面怎么样？'或者早上醒来说'你三周没联系 Thomas 了，我注意到他现在在镇上因为我看了他的 Instagram，要打个招呼吗？'"

更深层的：

> [01:28:43] "或者说'我注意到每次你见那个人你都会难过。为什么？'"

### "技术消失"的魔力 [01:30:55 - 01:31:12]

> [01:30:55] "你只是和手机上的一个朋友说话，它无限resourceful，有权访问你的邮件、日历、文件，能为你建网站、做行政工作、抓取网站、给朋友打电话或者给商家打电话预约。你不需要想复杂的东西，所有上下文都融合消失了。"

### 实际体验：摩洛哥之旅 [01:30:13 - 01:31:55]

他在摩洛哥朋友生日旅行时用 WhatsApp 和 agent 聊天：

> [01:30:25] "它带我游览城市，讲笑话，能代表我给其他朋友发 WhatsApp。"

**语音消息的惊艳时刻**：

他发了一条语音消息，没有告诉 agent 怎么处理。30 秒后 agent 回复了：

> [01:31:15] "你发给我一个文件，我看了 header 发现是 OGG 格式，用 FFmpeg 转换了，然后找 Whisper 但你电脑没装，但我找到了你的 OpenAI key，就 curl 到 OpenAI 服务器转录了。"

这是 Opus 4.5 的资源整合能力：

> [01:31:41] "它就是这么 resourceful。其他人说你需要 skill 或者 system，不，它就自己想出来了。"

### CLI 大军 vs MCP [01:32:55 - 01:35:39]

**为什么 CLI 而不是 MCP？**

> [01:33:08] "MCP 最好的事情是让公司愿意开放更多 API。但整个概念是蠢的。"

**MCP 的问题**：

1. 必须在 session 开始时预加载所有函数和解释
2. 不能链式调用、不能脚本化
3. 不能过滤——比如要 500 个城市列表，不能在 MCP 里只要伦敦的
4. 输出太多无关信息污染 context

**CLI 的优势**：

> [01:34:18] "模型非常擅长使用 bash。如果是 CLI，我可以用 jq 过滤只要我需要的。"

他建了一个 MCP Porter，把 MCP 转成 CLI。

### 自我更新的 Agent

> [01:39:27] "你可以字面上告诉你的 bot'update yourself'，它会 fetch 自己然后更新自己，然后回来说'我有新功能了'。"

## 十二、可落地的行动清单

### 技术实践

1. **闭环设计**：每个功能都要思考"agent 怎么自己验证这个"
2. **CLI 优先**：为你的工具建 CLI，而不是依赖 MCP
3. **本地测试**：让 agent 运行测试，不要等 CI
4. **并行 agent**：学会同时管理 5-10 个 agent，像下多盘棋
5. **参考其他项目**：prompting 时指向其他项目的解决方案

### 协作方式

1. **PR → Prompt Request**：在 PR 里附上 prompts
2. **讨论架构而非代码**：把时间花在设计讨论上
3. **功能重写而非 review**：用 agent 根据 PR 意图重建

### 工作流程

1. **对话式 prompting**：先讨论、给选项，再说"build"
2. **感受摩擦力**：注意 agent 的响应时间和 push back
3. **迭代塑形**：像雕塑一样，从石头里慢慢雕出形状
4. **forward-only**：很少回滚，更多是"那就改成这样"

### 学习建议

1. **保持无限好奇**
2. **多构建多尝试**——这是一项需要练习的技能
3. **学习系统架构**——不需要写每一行代码
4. **公开工作**——GitHub 活跃度是最好的简历
